# v0.5 Morrison Regression — Running Deltas

Per the plan's inline regression discipline, after each new rule lands we run it against
`packs/vv40/examples/morrison/cou1/uofa-morrison-cou1.jsonld` and
`packs/vv40/examples/morrison/cou2/uofa-morrison-cou2.jsonld` and record the delta here.

Baseline (`v0.4.0-nafems` tag):
- Morrison COU1: **14 weakeners** (W-EP-01×1, W-EP-02×3, W-AL-01×3, W-AR-05×3, COMPOUND-01×3, COMPOUND-03×1)
- Morrison COU2: **6 weakeners** (W-EP-04×6)
- Cross-COU divergences: 7

## W-ON-02 — Unbounded Applicability (High)

| | Count before | Count after | Delta | New firings |
|---|---|---|---|---|
| Morrison COU1 | 14 | 16 | +2 | `cou1-cpb` (W-ON-02, High) + cascade: +1 COMPOUND-01 (High/Critical pairing) |
| Morrison COU2 | 6 | 7 | +1 | `cou2-vad` (W-ON-02, High). No COMPOUND cascade — COU2 has no Critical weakener. |

**Interpretation:** Expected. Morrison examples were authored under v0.3/v0.4 before
`hasApplicabilityConstraint` and `hasOperatingEnvelope` were semantically required for a
clean COU. W-ON-02 is documenting a real gap in the assurance package, which is exactly
what the rule is meant to surface. No fix to the example data — let the weakener fire.

**Compound recount:** COU1's COMPOUND-01 count shifts +1 (3 → 4) because the new
High-severity W-ON-02 pairs with an existing Critical weakener through COMPOUND-01's
Critical+High cascade rule. COU2 unaffected (no Critical weakener on COU2).

## W-CON-05 — Activity-Evidence Consistency (High, Python)

| | Count before | Count after | Delta | New firings |
|---|---|---|---|---|
| Morrison COU1 | 16 | 16 | 0 | none |
| Morrison COU2 | 8 | 8 | 0 | none |

**Interpretation:** Morrison does not declare any activity directly on the UofA via
`uofa:hasVerificationActivity` (a new v0.5 property). Rule precondition cannot hold.

**Implementation note:** W-CON-05 was initially attempted as a two-stage Jena rule
(stage 1 marks activities that produced evidence; stage 2 checks noValue on that
marker). Forward-RETE evaluates stage 2's `noValue` before stage 1's derived
triples become observable, so stage 2 fires with false positives on activities
that DO have evidence. Moved to a Python post-pass
(`src/uofa_cli/python_rules/activity_evidence.py`) that walks the complete graph
once and avoids the ordering issue. The Jena rule is replaced by a comment
explaining the pitfall for future editors.

| | Count before | Count after | Delta | New firings |
|---|---|---|---|---|

the Python post-pass. The existing Jena COMPOUND-01 rule iterates only Jena-inferred
weakeners, so it does NOT re-fire on W-PROV-01 hits. This is a v0.5-scope limitation;
escalation across Python+Jena weakeners can be added in v0.6.

rule that silently passes on other Complete gaps.
