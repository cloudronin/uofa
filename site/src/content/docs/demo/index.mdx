This page reproduces the live demo presented at NAFEMS Americas 2026 on May 27, 2026. Same evidence packages, same pipeline, same rule engine, same numbers. The terminal output you see here is what the room saw on stage.

## What you will see

Morrison et al. (2019) is the FDA-co-authored worked example for ASME V&V 40 credibility assessment of a centrifugal blood pump. The same model, mesh, and validation data are evaluated under two different intended uses:

| COU | Intended use | Device class | Model Risk Level | Decision |
|---|---|---|---|---|
| COU 1 | Cardiopulmonary bypass support (short duration) | Class II | MRL 2 | Accepted |
| COU 2 | Ventricular assist device support (long duration) | Class III | MRL 5 | Not accepted |

The UofA pipeline takes the bundled Morrison evidence packages and produces two weakener reports. COU 1 fires <strong>{cou1.total} weakeners across {cou1.patterns} patterns</strong> with no compound risk. COU 2 fires <strong>{cou2.total} weakeners across {cou2.patterns} patterns</strong>, including {cou2.compound_hits} firings of `COMPOUND-01` (Critical and High coexistence).

## Reproduce in 10 minutes

This walkthrough runs the full authoring pipeline end-to-end against the bundled Morrison source folder, then exercises the rule engine on the generated JSON-LD. If you just want to see the rule engine without running the LLM extract, skip to stage 5 — the pre-built JSON-LD is also bundled.

Prerequisites: Python 3.10+, Java 17+. If Java is unavailable, use `--skip-rules` on `uofa check` and you will still get SHACL and integrity validation. For the `uofa extract` step, you need either a local Ollama install (default) or a hosted LLM provider configured via LiteLLM.

### 1. Clone and install

```bash
git clone https://github.com/cloudronin/uofa
cd uofa
pip install -e '.[excel]'
```

The bundled examples and the rule engine ship with the repo; no extra setup.

### 2. Extract structured evidence from prose (the LLM step)

The Morrison source folder is bundled with the repo:

```bash
ls packs/vv40/examples/morrison/source/
# EVIDENCE_MANIFEST.txt
# ansys_cfx_solver_settings.txt
# boundary_conditions_specification.txt
# cou_definitions.docx
# credibility_assessment_narrative.docx
# decision_rationale_cou1.pdf
# decision_rationale_cou2.pdf
# hemolysis_mih_comparison.csv
# mesh_convergence_study.csv
# monte_carlo_uq_cou2.csv
# piv_velocity_validation.csv
# risk_assessment_memo.pdf
```

Run the extract pipeline against the folder:

```bash
uofa extract packs/vv40/examples/morrison/source/ \
  --pack vv40 \
  --model ollama/qwen3.5:4b \
  -o morrison-extracted.xlsx
```

The default model is `ollama/qwen3.5:4b`, which runs locally and reports F1 = 1.00 against the Morrison ground truth. To use a hosted provider, substitute any LiteLLM-compatible model identifier (`gpt-4.1`, `claude-sonnet-4-7`, `gemini-2.5-pro`).

The output is a single `.xlsx` with one sheet per V&V 40 factor. No JSON-LD or RDF knowledge required.

### 3. Review the extracted xlsx (the human checkpoint)

This step is where the engineer catches what the LLM missed. `uofa extract` is fast, but it is not authoritative.

Open `morrison-extracted.xlsx` in Excel or LibreOffice. Walk through each sheet and confirm:

- The credibility factor levels match what the source documents claim, not what the LLM inferred
- Per-result acceptance criteria are present and correctly transcribed
- The COU classification (COU1 = CPB, COU2 = VAD) is assigned correctly
- The decision text in the Decision sheet matches the documented outcome

For the bundled Morrison source the LLM extracts at F1 = 1.00, so the review is fast. For your own evidence, expect to spend more time here. This is the design intent: the LLM accelerates the boring transcription work, the human owns the credibility judgment.

### 4. Import to signed JSON-LD

Generate a signing key:

```bash
mkdir -p keys
uofa keygen keys/morrison
```

Then import:

```bash
uofa import morrison-extracted.xlsx \
  --pack vv40 \
  --sign --key keys/morrison.key \
  --check
```

Expected: `Conforms` on SHACL, `signature valid` on integrity, ~30 seconds wall clock. The result is two signed JSON-LD packages, one per COU.

If you would rather skip the extract pipeline and use the pre-built JSON-LD shipped in the repo, the next two commands run against either the package you just produced or the canonical bundled version.

### 5. Run the rule engine — COU 1 (CPB use, MRL 2)

```bash
uofa rules packs/vv40/examples/morrison/cou1/uofa-morrison-cou1.jsonld --build
```

Expected: <strong>{cou1.total} weakeners detected</strong> across {cou1.patterns} patterns. Severity breakdown: {cou1.by_severity.Critical} Critical / {cou1.by_severity.High} High / {cou1.by_severity.Medium} Medium / {cou1.by_severity.Low} Low.

| Pattern | Severity | Hits |
|---|---|---|
| W-AR-05 | High | 3 |
| W-EP-02 | High | 3 |
| W-AL-01 | High | 3 |
| W-ON-02 | High | 1 |
| W-CON-04 | Medium | 1 |

Zero compound inferences. The CFD-side UQ omission is methodologically defensible at MRL 2 (Morrison's rationale is that bench testing carries the safety determination), but the rule engine still flags it as a potential reuse blocker at higher MRLs.

### 6. Run the rule engine — COU 2 (VAD use, MRL 5)

```bash
uofa rules packs/vv40/examples/morrison/cou2/uofa-morrison-cou2.jsonld
```

Expected: <strong>{cou2.total} weakeners detected</strong> across {cou2.patterns} patterns. Severity breakdown: {cou2.by_severity.Critical} Critical / {cou2.by_severity.High} High / {cou2.by_severity.Medium} Medium / {cou2.by_severity.Low} Low.

| Pattern | Severity | Hits |
|---|---|---|
| W-PROV-01 | Critical | 7 |
| ⚡ COMPOUND-01 | Critical | {cou2.compound_hits} |
| W-EP-04 | High | 6 |
| W-ON-02 | High | 1 |
| W-AL-02 | Medium | 1 |
| W-CON-04 | Medium | 1 |

`COMPOUND-01` fires when Critical and High weakeners coexist on the same UofA. It cannot be expressed as a single SPARQL query because it reasons about the *output* of the Level-1 rules. This compound inference is the central differentiator over query-based credibility checking.

### 7. Diff the two COUs

```bash
uofa diff packs/vv40/examples/morrison/cou1/uofa-morrison-cou1.jsonld \
          packs/vv40/examples/morrison/cou2/uofa-morrison-cou2.jsonld
```

Same model, same data, different intended use produces a measurably different credibility evidence profile. The diff surfaces every divergent pattern automatically.

### 8. (Optional) Cross-domain reproduction with NASA-STD-7009B

The same CLI runs against the HPT-blade thermal-analysis example with a different domain pack. The bundled HPT example (a turbomachinery conjugate-heat-transfer case study) ships at MRL levels where the rule engine reports a clean profile — substitute your own MRL-elevated case to exercise the rule engine on the aerospace pack.

```bash
uofa rules packs/nasa-7009b/examples/aerospace/uofa-aero-cou1-nasa7009b.jsonld --pack nasa-7009b
uofa rules packs/nasa-7009b/examples/aerospace/uofa-aero-cou2-nasa7009b.jsonld --pack nasa-7009b
```

The source evidence for the HPT example is bundled as zips alongside the JSON-LD (`aero-evidence-cou{1,2}.zip`) — unzip and point `uofa extract` at the directory to reproduce the authoring on-ramp shown above for Morrison.

## Run uofa on your own evidence

The pipeline above is the same one you would run against your own credibility evidence. Drop your source PDFs into a folder, point `uofa extract` at it, review the xlsx, import. No data leaves your environment — the CLI is fully local, and the optional `uofa extract` step supports local LLMs (Ollama qwen3.5:4b is the default).

See [Get Started](/start/) for installation details and the [Excel on-ramp](/start/from-excel/) for a fuller walkthrough of the import-side workflow.
